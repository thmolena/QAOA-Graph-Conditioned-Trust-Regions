# Source data for the GCTR manuscript

This directory contains the complete released outputs used to render the data
figures and tables. They were generated by `specops-gctr` 3.2.0 at seed
`20260424` with implementation SHA-256
`efe87496cbd928ff1ccd4371aea0ad3f4c5c84d8187f5e699f260b983b81265b`.
`meta.json`, `results.json`, and the portable figure manifest record schema 2,
the configuration, software environment, scoring semantics, implementation
fingerprint, and artifact hashes.

`generator-source-efe87496.zip` preserves the exact pre-refactor generator
snapshot: 15 Python modules under `code/specops_gctr/` and the
`code/pyproject.toml` used by the canonical simulation. Those paths describe
the frozen archive, not the live repository layout; the live package now resides
at root under `specops_gctr/` with one root `pyproject.toml`. The archive's
package fingerprint matches the `implementation_sha256` stored in `meta.json`
and `results.json`. The live package may contain later layout or display-only
fixes, so the manifest records generator and replot fingerprints separately.

The primary statistic is capped benchmark cost to a shared, privileged
per-instance reference. If a method reaches `0.98 * V_i`, its score is the
number of objective evaluations used; otherwise its score is the shared cap
of 400. `evaluations_used` remains the raw work performed, and
`reached_target` records attainment. This convention prevents an early-stopped
failure from looking cheap. It is not a deployable stopping rule unless a user
has an external reference value comparable to `V_i`.

The offline target generator uses eight local searches with a cap of 60 calls
each. The configured upper bounds are 115,200 objective calls for 240 training
targets, 38,400 for 80 validation targets, and 23,040 for 48 test references:
176,640 in total before the primary benchmark. Actual target-generation counts
were not persisted and can be smaller because a local search may converge
before its cap. The validation target arrays are carried through the data
object but are not consumed by the current model or calibrator; removing that
38,400-call upper-bound step and recording all offline counters are required
protocol improvements. Cross-size studies generate additional references.

The recorded environment is Python 3.13.12, NumPy 2.4.3, SciPy 1.17.1,
NetworkX 3.6.1, PyTorch 2.11.0, Matplotlib 3.10.8, and pandas 3.0.2 on
macOS/arm64 with one PyTorch thread. Runtime columns are machine-dependent.
Floating-point results can shift across PyTorch, BLAS, and platform builds.

No permanent archival DOI has yet been assigned. Before journal publication,
the final release should be deposited in a versioned archive such as Zenodo.

Author: Molena Huynh, North Carolina State University,
`molena.huynh@jmp.com`.

## Files

### `Figure2_QueryEfficiency.csv`

One aggregate row per primary method. `evaluations` and `evaluations_sd` are
the mean and population SD of capped cost; `evaluations_used` fields are raw
work; `successes` and `n_test` report attainment. Exact returned expectation
ratios, sampled best-bitstring ratios, machine-dependent runtime, and raw
two-sided paired Wilcoxon p-values versus GCTR are also included.

### `QueryEfficiency_InstanceLevel.csv`

All 48 test instances crossed with all six primary methods (288 rows), with
graph identity, shared target, capped score, raw use, attainment, exact
expectation ratio, and sampled ratio. These rows are the source for the
primary means, SDs, success counts, and paired tests.

### `Figure3_CalibrationAndUncertainty.csv`

The ten-level marginal coverage curve for 192 standardized angle residuals.
`covered_count` is cumulative at each nominal level, not a disjoint-bin count.

### `Figure4_Generalization.csv`

Cross-size aggregate capped costs and success counts for random restarts, the
fixed heuristic, the GNN point baseline, and GCTR. `speedup_vs_random` is only
the arithmetic ratio of mean capped costs; it is not a hardware speedup.

### `Figure5_Ablation.csv`

Aggregate capped cost, success count, marginal angle ECE, and difficulty-score
Spearman statistic per variant. Policy controls reuse the trained model;
loss/feature controls retrain it. The two-seed/full-cap control changes both
the Gaussian-seed allocation and the cap, so it is not a single-factor causal
ablation.

### `Figure7_BudgetPolicy.csv`

Per-instance standardized score `z`, nominal seed count `K_seeds`, and early
cap `T_budget_cap`. Under shared-cap miss scoring, an early cap cannot improve
capped cost along an otherwise identical trajectory; it can only tie or hurt.
The cap remains useful only as a raw-resource/quality tradeoff that this
benchmark does not identify by itself.

### `LOFO_InstanceLevel.csv`

All 48 leave-one-family-out test cases crossed with five policies (240 rows),
including family, graph seed, shared target, capped score, raw use, and
attainment. The four family-level fits induce clustering, so per-instance
Wilcoxon results are exploratory rather than independent replication.

### `EDFig2_ShotNoise.csv`

Finite-shot aggregate capped costs, noisy-stopping success counts, and exact
returned expectation ratios at 128 and 1024 shots. `successes` records the
optimizer's noisy target decision; it is not independently recomputed exact
attainment.

### `EDFig3_SeedStability.csv`

One row per algorithm seed and method on the same graph/target test set. This
measures training/search-seed sensitivity, not replication over new datasets.

### `results.json` and `meta.json`

Machine-readable full results and metadata. Primary and LOFO per-instance rows
are retained; returned parameter vectors and per-instance rows for every
auxiliary study are not. The full command regenerates those studies from
source, but the committed aggregates alone do not reconstruct every trajectory.
